# Flagship Showcase PR Plan

This document supersedes the earlier productization-oriented backlog sequence
for the current goal of curating the repository as a flagship platform
showcase.

## Planning Rules

1. Optimize for showcase clarity, demo reliability, and credible engineering
   signals.
2. Keep breadth; do not remove modules just to simplify the narrative.
3. Treat tours, seeded data, smoke paths, and operator polish as first-class
   deliverables.
4. Use release and governance docs as the safety rails for showcase evolution.

## PR Sequence

| PR | Title | Outcome |
| --- | --- | --- |
| `PR-01` | Showcase Taxonomy and Tour Map | audience-first framing |
| `PR-02` | Demo Contract and Setup Guardrails | one supported hero path |
| `PR-03` | Demo Seed, Reset, and Recovery | deterministic showcase state |
| `PR-04` | Frontend Workspace Quality Gates | real frontend health coverage |
| `PR-05` | Typed Federation and Shared Edges | cleaner shell-to-remote story |
| `PR-06` | AI Monitor Polish and Operator Tour | stronger operator exhibit |
| `PR-07` | Core Showcase Smoke Paths and CI Tiers | CI reflects flagship priorities |
| `PR-08` | Architecture Diagrams and Evidence Pack | faster reader comprehension |
| `PR-09` | Showcase Stewardship and Review Cadence | sustainable curation model |

## Acceptance Themes

- every official tour is documented and reproducible
- hero paths are easy to start and hard to break silently
- optional modules remain discoverable without becoming blockers
- operator tooling feels intentional rather than incidental
- docs and CI explain the platform the same way
